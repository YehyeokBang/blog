# 프로젝트 개요

## 배경

Velog를 운영하며 **플랫폼 종속에 대한 불안감**과 **관리 부실에 대한 실망감**을 느꼈다.
"내가 쓴 글이 영원히 내 것이었으면" 하는 바람과, 백엔드 엔지니어로서 **SEO, 트래킹, 분석** 감각을 키우고 싶은 욕구가 만났다.

- 기존 Velog 운영 현황: 42개 포스트, 팔로워 140명
- 팔로워/팔로잉은 포기해도 괜찮음
- Velog와 병행 운영 가능 (필요시)

## 핵심 가치 (우선순위)

1. **내 글의 영속성** — Markdown 파일이 정본. 플랫폼이 사라져도 글은 남는다.
2. **백엔드 실험실** — 조회수, 검색, 좋아요 등을 직접 구현하며 배우는 장난감.
3. **트래킹 & 분석 감각** — GA4와 Search Console로 검색 유입과 실제 읽기 행동을 관찰.
4. **읽기 좋은 경험** — 글의 깊이에 맞는 구조, 편안한 타이포그래피.

## 프로젝트 성격

- **개인 블로그**: 나만 글을 올린다. 다른 사용자의 글 업로드는 고려하지 않는다.
- **프론트엔드 + 백엔드 분리 아키텍처**: 백엔드는 "실험실"로 활용한다.
- **MVP 속도보다 방향성 우선**: 빠른 출시가 목표가 아니라, 만드는 과정 자체가 학습이다.
- **Docker 기반 배포**: 약 4GB 메모리의 클라우드 인스턴스에 컨테이너로 배포.

## 콘텐츠 식별 규칙 (파일명 불변 법칙)

- **파일명(slug)은 고유 식별자(PK)로 작동**: 마크다운 파일명(`slug`)은 향후 연동할 백엔드 데이터베이스의 조회수, 좋아요, 댓글 등의 테이블에서 각 글을 매핑하는 고유 Primary Key 역할을 수행합니다.
- **이미 배포된 마크다운 파일명 변경 금지**: 이미 프로덕션 환경에 배포되어 조회수나 댓글 데이터가 쌓인 포스트의 파일명은 원칙적으로 수정하지 않습니다. 
- **예외적 변경 처리**: 불가피하게 파일명을 수정해야 할 경우에는 백엔드 데이터베이스(SQLite 등)의 관련 테이블 내 `slug` 레코드를 수동(SQL 쿼리 직접 수행)으로 업데이트하여 데이터 정합성을 유지합니다.
