# 📝 방예혁 개인 블로그 프로젝트 기획서

본 기획서는 개인 블로그 프로젝트의 개발 목적, 타겟 독자, 디자인 방향성, 핵심 기능 로드맵 및 아키텍처를 종합적으로 정리한 문서입니다. 디자인(UI/UX) 작업 및 프론트엔드/백엔드 개발의 나침반 역할을 합니다.

---

## 1. 프로젝트 개요 및 배경

### 1.1 기획 배경
- **플랫폼 종속성 탈피**: 기존 Velog에서 42개의 포스트를 운영하며 플랫폼 종속성에 대한 불안감과 관리의 한계를 체감함.
- **데이터 소유권**: "내가 쓴 글이 영원히 내 것이었으면" 하는 목표로, Markdown 파일 기반의 영속적인 블로그 시스템이 필요함.
- **백엔드 실험실**: 주니어 백엔드 엔지니어로서 실서비스 환경에서 SEO, 트래킹, 동시성 처리(조회수, 검색 등)를 직접 구현해보고 실험할 수 있는 사이드 프로젝트가 필요함.

### 1.2 핵심 가치 (우선순위)
1. **내 글의 영속성**: Markdown 파일이 정본(SSoT)이 되어, 플랫폼이 사라져도 글 데이터는 안전하게 보존됨.
2. **백엔드 실험실**: 단순한 정적 블로그를 넘어 백엔드 기술(조회수, 좋아요, 검색 등)을 실험할 수 있는 아키텍처.
3. **트래킹 & 분석**: GA4 및 Firebase Analytics를 활용해 사용자 행동을 직접 관찰하고 데이터 기반의 인사이트 확보.
4. **쾌적한 읽기 경험**: 기술 글의 깊이에 맞는 넉넉한 여백, 편안한 타이포그래피, 가독성 높은 디자인 제공.

## 2. 타겟 독자 및 페르소나

### 2.1 블로그 주인 (운영자)
- **이름**: 방예혁 (YehyeokBang)
- **직업**: 주니어 백엔드 엔지니어 (Java/Kotlin + Spring Boot 주력)
- **글 성향**: 단순 TIL(Today I Learned) 지양. 문제 상황을 정의하고, 이를 해결하기 위한 탐색 과정과 구현/검증(코드, 수치 등), 그리고 회고까지 깊이 있게 서술하는 스토리텔링형 에세이 및 기술 블로그 지향. (글 길이 2,000~5,000자 수준)

### 2.2 타겟 독자
- **주 독자**: 비슷한 기술적 고민(동시성, 성능 최적화, 쿼리 튜닝 등)을 하는 동료 개발자.
- **부 독자**: 포트폴리오를 검토하는 채용 담당자.
- **독자 니즈**: 단편적인 정답보다는 "왜 이런 기술을 선택했는지"에 대한 논리적 사고 과정과 생생한 경험담.

## 3. 디자인 방향성 (톤 앤 매너)

디자인은 개발자 블로그의 표준을 따르되, 지루하지 않고 고급스러운 느낌을 주어야 합니다.

- **전체 무드**: 깔끔하고 전문적이며 모던한 느낌 (GitHub의 안정감 + Toss의 세련됨).
- **여백과 타이포그래피**: 글 자체에 집중할 수 있도록 여백을 넉넉하게 사용하고, 가독성이 뛰어난 폰트와 행간 적용.
- **인터랙션**: 미니멀리즘을 유지하되 밋밋함을 없애기 위한 미묘하고 부드러운 애니메이션 적용.
- **필수 UI/UX 요소**:
  - **다크/라이트 모드**: 다크 모드를 기본으로 하되 사용자 선택 토글 제공.
  - **TOC (목차) 사이드바**: 긴 글 읽기 경험을 돕기 위해 데스크탑 환경에서 우측/좌측 고정된 목차(현재 위치 하이라이트).
  - **스크롤 프로그레스 바**: 최상단에 스크롤 진행도를 나타내는 인디케이터.
  - **코드 블록**: 개발자 블로그의 핵심인 깔끔한 Syntax Highlighting과 원클릭 '복사 버튼'.
  - **네비게이션**: 글의 맥락을 이어주는 시리즈 내 '이전/다음 글' 이동 UI.
  - **목록 뷰**: 최신순 중심의 피드형 글 목록 및 태그별 필터링 기능.

## 4. 단계별 기능 로드맵

### Phase 1: MVP (프로토타입)
프론트엔드 중심으로 백엔드 없이 빠르게 동작하는 정적(SSG) 블로그 구축.
- **페이지**: 피드형 글 목록, Markdown 기반 상세 글 페이지.
- **핵심 기능**: Markdown 렌더링, 태그 필터링, 다크/라이트 모드, TOC, 코드 하이라이팅.
- **마케팅/운영**: SEO 최적화 (메타 태그, OG 이미지, Sitemap), GA4 연동(기본 페이지뷰).
- **환경**: 반응형 웹, Docker 기반 배포.

### Phase 2: 백엔드 실험실 가동
Kotlin + Spring Boot 백엔드를 추가하여 동적 기능 구현 (본격적인 실험실).
- 조회수 트래킹 API (동시성 제어 학습)
- 제목/본문 전문 검색 API
- 좋아요 기능 (비회원, IP/쿠키 기반)
- Firebase Analytics 활용 (이탈률, 스크롤 깊이 등 상세 트래킹)
- 시리즈 글 그룹핑 및 글 하단 관련 글 추천 기능

### Phase 3: 기능 고도화
- Markdown WYSIWYG 웹 에디터 도입
- 댓글 시스템 (자체 구현 또는 GitHub 기반 연동)
- 이미지 전용 업로드 기능 및 CDN 연동
- 기존 Velog 글 자동 마이그레이션 도구

## 5. 아키텍처 및 시스템 구조

- **Frontend**: Next.js (TypeScript, React) — 정적 사이트 생성(SSG)을 통한 SEO 최적화 및 빠른 초기 로딩 속도 달성.
- **Backend**: Kotlin + Spring Boot — 익숙한 주력 기술로 API 서버 구축.
- **Database**: SQLite (Phase 1, 2) — 4GB 메모리 제약을 고려해 별도 프로세스가 불필요한 SQLite 채택. 향후 동시성 문제 발생 시 PostgreSQL로 마이그레이션.
- **인프라/배포**: 단일 4GB 클라우드 인스턴스 위에서 Docker Compose를 이용해 프론트엔드/백엔드를 컨테이너 기반으로 분리 배포. CI/CD는 GitHub Actions 활용.

### 5.1 콘텐츠 관리 (하이브리드 CQRS 패턴)
- **Write Path**: IDE에서 Markdown 파일 작성 후 Git에 Push 시 배포 트리거 동작. Next.js가 파싱 및 SSG 빌드를 수행하고, 백엔드는 DB에 메타데이터를 동기화. **(Markdown 파일이 절대적 원본)**
- **Read Path**: 사용자는 Next.js에서 빌드된 정적 HTML을 서빙받고, 조회수/검색 등 동적 데이터만 백엔드 API를 통해 채움.
