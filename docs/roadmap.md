# 제품 로드맵

## 북극성

검색을 통해 발견된 독자가 콘텐츠를 의미 있게 읽고, 운영자가 부담 없이 글과 작은 기능 실험을 계속 추가할 수 있는 개인 콘텐츠 자산을 만든다.

조회수 자체보다 **검색 유입 후 의미 있게 읽힌 비율과 추세**를 우선한다. 의미 있는 읽기의 이벤트 조건과 수치 목표는 GA4와 Search Console 기준선을 확보한 뒤 정한다.

## 운영 방식

- 고정 마감이나 월간 발행·개발 수량을 두지 않는다.
- 작은 backlog를 유지하고 한 번에 하나씩 꺼내 끝낸다.
- 완성된 작은 변경은 운영에 배포해 실제 반응을 관찰한다.
- 다음 항목은 이전 항목의 관찰 결과와 유지 부담을 확인한 뒤 선택한다.

## Now

1. **측정 기반 확립**
   - GA4 주입 상태와 production 데이터를 확인한다.
   - Search Console 소유권과 sitemap 수집 상태를 확인한다.
   - 검색 유입, 읽기 깊이, 이탈을 함께 볼 수 있는 최소 기준선을 만든다.
2. **의미 있는 읽기 이벤트**
   - 기준선에서 필요한 이벤트 조건을 정하고 과도한 이벤트 수집을 피한다.
   - 페이지뷰와 실제 읽기를 구분할 수 있게 한다.
3. **조회수 API**
   - Markdown slug를 식별자로 사용한다.
   - 단순한 첫 버전을 운영에 배포해 중복 집계와 부하 특성을 관찰한다.

## Next

- **소유자 전용 대시보드**: 공개 화면을 복잡하게 만들지 않고 핵심 유입·읽기·조회 지표를 운영자가 확인한다.
- **댓글 관리**: 이미 제공 중인 공개 댓글의 부적절한 내용을 운영자가 숨기거나 삭제할 수 있게 한다.

## Later

- **검색과 콘텐츠 탐색**: 제목·본문 검색과 태그·관련 글 탐색을 실제 콘텐츠 규모에 맞게 개선한다.
- **작은 AI 실험**: 요약, 탐색 보조 등 측정 가능한 한 가지 문제만 골라 작게 검증한다.

## 기술 작업 우선순위

서로 경쟁하는 기술 작업은 다음 순서로 판단한다.

1. 신규 아이디어를 가장 작게 검증할 수 있는가
2. 기존 콘텐츠와 사용자 데이터를 안전하게 보존하는가
3. 노출된 보안 위험을 줄이는가
4. 실제 변경을 방해하는 기술 부채를 줄이는가
5. 측정된 성능 병목을 개선하는가

이 순서는 데이터 보존이나 보안을 희생해 기능을 출시한다는 뜻이 아니다. 각 변경의 안전 조건을 충족한 뒤, 추가 투자 순서를 정하는 기준이다.

## 출시와 검증

- 기본 방식은 작은 변경을 production에 배포하고 관찰하는 것이다.
- DB schema·mount·식별자처럼 데이터 보존에 영향을 주거나 복구가 어려운 변경은 별도 환경과 복구 절차로 먼저 검증한다.
- rollback 또는 기능 비활성화 방법이 없는 변경은 범위를 더 작게 나눈다.

## 최상위 위험

가장 큰 위험은 기능 부족이 아니라 **복잡성과 유지 부담 때문에 프로젝트를 더 이상 가꾸지 않게 되는 것**이다. 새 도구와 추상화는 현재 backlog 하나를 끝내는 데 꼭 필요한 경우에만 추가한다.
